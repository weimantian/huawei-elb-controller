# 华为云 CCE 上 MongoDB / PostgreSQL / MySQL DBC 创建根因分析报告

> 日期: 2026-07-14（初版），2026-07-14（实测刷新）
> 分析范围: Huawei ELB Controller + OpenEverest (`percona/everest-operator`) 1.16.1
> 验证环境: CCE cn-north-4a, K8s v1.35.3-r0-35.0.8, 3 节点 s6.xlarge.2

---

## 一、概述

在华为云 CCE 集群中，OpenEverest 创建 DatabaseCluster (DBC) 时，`LoadBalancer` 类型的 Service 默认无法获取 `EXTERNAL-IP`，因为 OpenEverest 上游不感知 CCE。

本报告从 `huawei-elb-controller` 与 `percona/everest-operator` 两个代码库的源码层面进行分析，并以 CCE 集群上 MySQL (PXC) / PostgreSQL (PG) / MongoDB (PSMDB) 三种引擎的端到端实测结果逐条验证。

| 代码库 | 版本/分支 | 说明 |
|---|---|---|
| `huawei-elb-controller` | 当前工作区 | 华为云 ELB 控制器，负责注入 CCE LoadBalancer 注解 |
| `percona/everest-operator` | 1.16.1 | OpenEverest，管理 DatabaseCluster 生命周期 |

## 三、OpenEverest 上游源码分析

### 3.1 注解传递链路 - 三引擎共用同一逻辑

所有三个引擎的 LoadBalancer Service 注解通过以下链路获取：

```
DatabaseCluster CR (spec.proxy.expose.type=LoadBalancer)
  ↓
everest-operator providers/{pxc,psmdb,pg}/applier.go
  ↓
common.GetAnnotations(ctx, client, database)    ← PXC/PSMDB/PG 三引擎共用
  ↓
GetLoadBalancerConfig() -> 查询 LoadBalancerConfig CRD
```

**核心代码** (`common/helper.go`):

```go
func GetAnnotations(ctx, c, database) (map[string]string, error) {
    lbc, err := GetLoadBalancerConfig(ctx, c, database)
    if err != nil {
        if errors.Is(err, ErrEmptyLbc) {
            return map[string]string{}, nil  // ← 无 LBC -> 返回空 map！
        }
        return nil, err
    }
    return lbc.Spec.Annotations, nil
}
```

**关键发现**: 如果用户创建 DatabaseCluster 时未指定 `loadBalancerConfigName`，`GetAnnotations()` 直接返回**空 map**。即使指定了，也只是读取 `LoadBalancerConfig.Spec.Annotations` 中的纯键值对，**不含任何云平台智能逻辑**。

### 3.2 集群类型检测 - 仅支持 EKS

**`consts/consts.go`**:
```go
ClusterTypeEKS      ClusterType = "eks"
ClusterTypeMinikube ClusterType = "minikube"
```

**`common/helper.go`** - `GetClusterType()`:
```go
func GetClusterType(ctx, c) (ClusterType, error) {
    clusterType := ClusterTypeMinikube  // ← 默认回退到 minikube
    for _, storage := range storageList.Items {
        if strings.Contains(storage.Provisioner, "aws") {
            clusterType = ClusterTypeEKS  // ← 只检测 AWS/EBS
            break
        }
    }
    return clusterType, nil
}
```

**在华为云 CCE 上，此函数永远返回 `minikube`**。OpenEverest 中没有 `cce`、`huawei` 或任何与之相关的常量或逻辑。

## 四、CCE Cloud Controller 的要求

CCE 的 Cloud Controller Manager (CCM) 在处理 `type: LoadBalancer` 的 Service 时，**必须**存在以下注解之一才会采取行动：

| 注解 | 必要性 | 值示例 | 谁负责生成 |
|---|---|---|---|
| `kubernetes.io/elb.class` | **必需** | `union` 或 `performance` | 控制器注入 |
| `kubernetes.io/elb.autocreate` | 必需（自动创建时） | JSON: `{"type":"public","bandwidth_name":"..."}` | 控制器注入 |
| `kubernetes.io/elb.id` | 可选（绑定已有 ELB） | ELB UUID | CCM 回写 |

**没有这些注解 -> CCM 完全忽略 -> `EXTERNAL-IP` 永久 `<pending>`**

---

## 五、Huawei ELB Controller 的角色

Huawei ELB Controller 正是为填补上述缺口而构建。其核心流程：

```
LoadBalancer Service 被创建（由 everest-operator 生成，无 CCE 注解）
  ↓
Huawei ELB Controller watch 到新 Service
  ↓
isOpenEverestService() -> 检查 managed-by label 匹配
  ↓
reconcileCreate() -> NetworkDetector.Detect() 获取 VPC/子网/AZ
  ↓
buildELBAnnotations() -> 生成 elb.autocreate + elb.class
  ↓
patchService() -> 注入注解到 Service
  ↓
CCE CCM 检测到注解 -> 创建 ELB -> EXTERNAL-IP 分配 ✅
```

### 5.1 OpenEverest Service 识别机制（实测验证）

```go
// service_utils.go
func isOpenEverestService(svc) bool {
    managedBy := svc.Labels["app.kubernetes.io/managed-by"]
    return openeverestOperators[managedBy]
}

var openeverestOperators = map[string]bool{
    "percona-xtradb-cluster-operator": true,
    "percona-server-mongodb-operator": true,
    "percona-postgresql-operator": true,
}
```

**实测验证**: OpenEverest 1.16.1 三个引擎创建的 Service label 全部匹配成功，controller 日志显示 3 引擎均成功注入注解。

## 六、各引擎端到端分析（实测刷新）

### 6.1 MySQL / PXC - ✅ 已验证可用

- **Service 架构**: 单个 HAProxy LoadBalancer Service（primary）+ 单个 HAProxy LoadBalancer Service（replicas）= 2 个 ELB
- **ELB Controller 处理**: 注入 2 组注解到 2 个 Service
- **CCM 行为**: 创建 2 个 ELB，分配 EXTERNAL-IP
- **客户端连接**: 直接连接 primary ELB IP 即可
- **测试状态**: E2E 通过（MySQL 8.4.7-7.1 连接验证成功）

### 6.2 PostgreSQL - ✅ 已验证可用

- **Service 架构**: 单个 PGBouncer LoadBalancer Service = 1 个 ELB
- **ELB Controller 处理**: 注入 1 组注解到 1 个 Service
- **CCM 行为**: 创建 1 个 ELB，分配 EIP `113.44.249.223`
- **客户端连接**: 连接 ELB IP，PGBouncer 内部负载均衡
- **测试状态**: E2E 通过（PostgreSQL 18.4 - Percona Server 18.4.1 连接验证成功）

### 6.3 MongoDB / PSMDB - ✅ 已验证可用（有架构特性需注意）

#### 6.3.1 非分片模式的 per-pod Service 架构（实测确认）

`percona-server-mongodb-operator` 在非分片模式下，为副本集的每个成员创建**独立的 LoadBalancer Service**:

```
Service 命名格式: {cr-name}-{replset}-{pod-index}
```

3 节点副本集实测：
```
mongodb-e2e-test-rs0-0   (LoadBalancer)  EIP: 121.36.97.223
mongodb-e2e-test-rs0-1   (LoadBalancer)  EIP: 120.46.221.33
mongodb-e2e-test-rs0-2   (LoadBalancer)  EIP: 113.44.249.223
```

对比：

| 引擎 | 外部 Service | ELB 数量 | 客户端拓扑 |
|---|---|---|---|
| PXC | 2 (primary+replicas HAProxy) | 2 | 直连 primary ELB IP |
| PG | 1 (PGBouncer) | 1 | 直连 ELB IP |
| **PSMDB 非分片** | **N（节点数）** | **N** | 需感知 Primary/Secondary |
| PSMDB 分片 | 1 (mongos) | 1 | 直连 ELB IP |

#### 6.3.2 多 ELB 的影响分析（实测刷新）

| 报告初版预测 | 实测结果 |
|---|---|
| 多 ELB 成本浪费 | ✅ 确认，3 节点 = 3 个 ELB。但这是 PSMDB operator 架构设计，非 controller bug |
| MongoDB 驱动拓扑发现失败 | ⚠️ 部分确认：副本集成员地址用 ClusterDNS 域名，外部客户端无法解析。**但这是 MongoDB 副本集 + LB 的通用问题，EKS/GKE 行为一致** |
| Controller reconcile 竞争条件 | ❌ **预测错误**：3 个 service 按 pod 启动顺序逐个处理（间隔 6-7 分钟），无并发竞争 |

#### 6.3.3 副本集形成实测（报告未覆盖的新发现）

实测发现 PSMDB operator 1.16.1 的一个行为问题：

**现象**：PSMDB operator 在创建完 admin 用户后，reconcile 循环卡住，不执行 `rs.add()` 添加 rs0-1/rs0-2 到副本集。

**证据**：
- operator 日志停在 `Creating user databaseAdmin`（10:40:21 UTC），之后无任何日志
- `rs.conf()` 显示副本集只有 rs0-0 一个成员
- 手动 `rs.add("rs0-1...")` 和 `rs.add("rs0-2...")` 后，副本集立即形成（PRIMARY + 2 SECONDARY）

**结论**：这是 PSMDB operator 的 bug，与 huawei-elb-controller 无关。手动 `rs.add()` 可恢复，或使用更新版本 operator。

#### 6.3.4 推荐的解决方案

| 方案 | 说明 | 工作量 | 推荐度 |
|---|---|---|---|
| **方案 A: 仅支持 PSMDB 分片模式** | 分片模式使用 mongos，只有 1 个 Service，架构与 PXC/PG 一致 | 低 | ⭐⭐⭐ |
| 方案 B: 非分片模式 + 手动 rs.add() | 创建后手动添加副本成员，外部通过任一 ELB 连接 | 中 | ⭐⭐ |
| 方案 C: 改造 Controller 支持多 Service | 支持一个 DBC 对应多个 ELB，需处理拓扑同步 | 高 | ⭐ |

## 九、结论

### 根本原因

**OpenEverest 上游完全没有 CCE 支持**。它仅内置 AWS/EKS 的集群类型检测和 NLB 自动创建逻辑，在 CCE 上所有引擎的 LoadBalancer Service 都无法获取 EXTERNAL-IP。

Huawei ELB Controller 通过注入 CCE 所需的注解来填补这个缺口，且**代码对各引擎的 Service 处理逻辑完全相同**（遍历所有 LoadBalancer Service，匹配 managed-by label，注入注解）。

### controller 代码无 bug

代码审查 + 三引擎端到端实测均未发现 controller 层面的 bug。报告初版预测的 managed-by label 不匹配（R3）、NetworkDetector 标签缺失（R4）均未发生。

### 实测发现的真实问题

| 问题 | 归属 | 影响 controller |
|---|---|---|
| PSMDB operator 不执行 rs.add() | PSMDB operator bug | 无 |
| CCM EIP 配额满时无效重试 | CCM 行为 | 无 |
| MongoDB 副本集 + LB 拓扑发现 | MongoDB 通用问题 | 无 |

### 推荐行动

1. ✅ **已完成**: PXC/PG/PSMDB 三引擎 E2E 测试全部通过
2. ✅ **已完成**: 确认 1.16.1 版本 managed-by label 匹配
3. ⚠️ **可选**: PSMDB 推荐使用分片模式（单 mongos Service，架构更简洁）
4. ⚠️ **可选**: 向华为云反馈 CCM 在 EIP 配额满时的重试行为优化建议
